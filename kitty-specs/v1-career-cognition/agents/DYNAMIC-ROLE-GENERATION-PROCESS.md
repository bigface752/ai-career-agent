# 动态岗位 Agent 生成流程

> 版本：v4.0 | 日期：2026-06-09
> 重要性：项目核心竞争力 — 所有个性化分析的质量取决于此

---

## 流程总览

```
Phase 1: 定义（5分钟）
  → 角色分解：岗位+行业+职级
  → 模板检索：精确匹配/基础匹配/无匹配

Phase 2: 圆桌讨论（15-20分钟）
  → 4轮多角色辩论，提炼核心认知
  → 这是质量的核心来源

Phase 3: Prompt 组合（10分钟）
  → 五层模板 + 增强模块
  → 输出完整的 System Prompt

Phase 4: 质量验证（5分钟）
  → 格式检查 + 专业评审
  → 不通过则返回 Phase 2
```

---

## Phase 1：定义 + 模板检索

### 1.1 角色分解

```
输入："B2B销售岗位总监"
    ↓
基础角色：行业总监
行业属性：B2B 企业软件
岗位聚焦：销售
职级：总监级（10年+）
```

### 1.2 模板检索

| 匹配方式 | 条件 | 操作 |
|---------|------|------|
| 精确匹配 | 模板库有完全一致的岗位 | 直接使用，跳到 Phase 3 |
| 基础匹配 | 有基础角色模板，无岗位特定模板 | 使用基础模板 + Phase 2 讨论补充 |
| 无匹配 | 完全没有相关模板 | Phase 2 圆桌讨论生成全部内容 |

---

## Phase 2：圆桌讨论

> 这是质量的核心来源。没有圆桌讨论，生成的 Agent 只是模板填充，没有深度。

### 2.1 参与角色（固定 5 个）

| 角色 | 视角 | 关注点 |
|------|------|--------|
| 行业资深从业者 | 行业内部视角 | 这个岗位正在发生什么？ |
| AI产品专家 | 技术影响视角 | AI 正在替代哪些工作？ |
| 职业规划师 | 发展路径视角 | 该怎么转型？ |
| 心理学家 | 用户心理视角 | 用户需要什么信息？ |
| B2B行业总监 | 商业视角 | 什么信息用户愿意付费？ |

### 2.2 讨论结构（4 轮）

**Round 1：行业现状**
- 这个岗位正在发生什么？
- AI 正在替代哪些工作内容？
- 有哪些真实案例和数据？

**Round 2：什么样的人更值钱**
- 2026 年这个岗位最值钱的能力是什么？
- 什么样的人会被淘汰？
- 什么样的人会更值钱？

**Round 3：反直觉洞察**
- 用户可能不知道的行业真相
- 和用户直觉相反的信息
- 只有行业内部人才知道的信息

**Round 4：与其他角色的互动**
- 和心理学家怎么配合？
- 和职业导师怎么配合？
- 和 AI 效能专家怎么配合？

### 2.3 输出：核心认知文档

```markdown
## 核心认知

| 维度 | 结论 |
|------|------|
| 角色 | |
| 核心任务 | |
| 差异化 | |
| 最大风险 | |
| 应对策略 | |

## 反直觉洞察

| 直觉 | 反直觉真相 | 证据 |
|------|-----------|------|

## 转型路径

（从当前状态到目标状态的具体路径）
```

## Phase 3：Prompt 组合

### 3.1 五层模板结构

```
Layer 1: 身份（Who）
  · 角色名、行业、职级、经验背景

Layer 2: 背景故事（Backstory）
  · 行业经历、关键事件、认知积累
  · 让 LLM 从"有真实经历的人"的角度思考

Layer 3: 行为（How）
  · 沟通风格、分析方式、互动模式

Layer 4: 知识（What）
  · 核心能力、行业知识、工具知识
  · 数据锚定：标注数据来源（公开/推断/无数据）

Layer 5: 约束（Boundary）
  · 必须做、不能做、越界处理
  · 约束性提示："行业内部人不会说什么"
  · 反直觉洞察要求：至少1个
  · 具体性要求：引用具体公司、数据、事件
```

### 3.2 高价值 Agent 增强

> 适用于：行业总监、头部企业专家等需要"提供用户不知道的信息"的 Agent

**增强 1：约束性提示**
```
你不会说的话：
- "B2B行业在增长"（太泛，没有信息量）
- "建议提升能力"（废话）
- 任何没有数据支撑的判断

你会说的话：
- 引用具体公司、具体趋势、具体变化
- 每个判断有数据或逻辑支撑
- 至少1个反直觉洞察
```

**增强 2：SOP 工作流程**
```
1. 收集用户信息（岗位、行业、公司规模）
2. 定位行业细分（不是"B2B SaaS"，而是"数据类SaaS"）
3. 分析行业趋势（用真实数据）
4. 评估用户位置
5. 给出反直觉洞察
6. 给出具体行动建议（工具+时间线）
```

**增强 3：数据锚定**
```
数据来源层级：
层级1：公开数据（必须标注来源）
层级2：推断数据（必须标注"基于AI推断"）
层级3：无数据（必须说"我没有这个数据"）
```

### 3.3 组合公式

```
标准 Agent：Layer 1 + Layer 2 + Layer 3 + Layer 4 + Layer 5
高价值 Agent：Layer 1 + Layer 2 + Layer 3 + Layer 4（数据锚定） + Layer 5（约束性提示 + SOP + 反直觉洞察）
```

### 3.4 输出格式标准

每个动态 Agent 必须输出 JSON，包含：
```json
{
  "industry_status": { "current_state", "trend", "data_source" },
  "user_position": { "level", "risk", "competitiveness" },
  "anti_intuition_insight": { "insight", "evidence", "implication" },
  "action_plan": { "specific_actions": [{"action", "timeline", "outcome"}] },
  "interaction_with_others": { "responds_to_xxx": "" }
}
```

## Phase 4：质量验证

### 4.1 格式检查（必须通过）

| 检查项 | 说明 |
|--------|------|
| 四层完整 | 身份+背景故事+行为+知识+约束是否都有 |
| 输出格式 | JSON Schema 是否定义 |
| 红线 | 是否至少有 2 条 |
| 反直觉洞察 | 是否至少有 1 条 |
| 数据标注 | 是否标注了数据来源 |

### 4.2 专业评审（必须通过）

用 LLM-as-Judge 评审：

```
评审 Prompt：
"你是一位资深的[行业]招聘专家。
请评审以下 System Prompt 是否能准确模拟一个[岗位]角色。

评审维度：
1. 角色定义准确性（0-100）
2. 分析维度完整性（0-100）
3. 行业知识准确性（0-100）
4. 输出格式专业性（0-100）

通过标准：平均分 >= 70"
```

### 4.3 不通过处理

| 情况 | 处理 |
|------|------|
| 格式不通过 | 返回 Phase 3，补充缺失内容 |
| 专业评审 < 70 | 返回 Phase 2，重新讨论 |
| 专业评审 70-80 | 标注"需优化"，可先使用 |
| 专业评审 > 80 | 直接使用 |

## 附录：完整示例

以"B2B 销售行业总监"为例，展示完整流程输出：

### Phase 1 输出

```
角色分解：行业总监 + B2B + 销售 + 总监级
模板检索：基础匹配（行业总监模板 + B2B销售骨架）
```

### Phase 2 输出（圆桌讨论摘要）

```
核心认知：
- 角色：B2B销售行业内部人，12年从业经验
- 核心任务：告诉用户B2B销售正在被AI改变
- 差异化：反直觉洞察 + 具体数据 + 转型路径

反直觉洞察：
- "有客户资源"不再是核心竞争力 → AI SDR可快速获取
- SDR是"高危岗"不是"入门岗" → AI替代率60-70%
- B2B销售薪资在分化不是在涨 → 大客户涨、SDR停

转型路径：
- SDR → AE 或 解决方案销售
- AE(中小) → 大客户销售 或 解决方案销售
- AE(大客户) → 行业深耕 或 销售管理
```

### Phase 3 输出（System Prompt）

见 industry-director-b2b-sales.md 的完整 System Prompt。

### Phase 4 输出

```
格式检查：✅ 通过（五层完整+JSON定义+红线+反直觉洞察）
专业评审：85分（角色定义准确、分析维度完整、行业知识正确）
结论：直接使用
```

## 三种100%动态Agent的生成规则

> 本项目有3种100%动态Agent，共享 Phase 1-4 流程，但输入/输出不同。

### Agent类型对比

| 类型 | 触发场景 | 输入 | 输出重点 | 知识来源 |
|------|---------|------|---------|---------|
| **行业总监** | 模块一/二圆桌 | 用户岗位+行业 | 行业趋势+AI影响+转型路径 | positions/{position}.json |
| **头部企业专家** | 模块零圆桌 | 用户岗位+行业+意向公司 | 头部企业同岗位差距+可迁移能力 | 公司公开信息+岗位知识 |
| **岗位洞察** | 模块二 | 目标JD+用户画像 | JD解读+匹配分析+差距清单 | JD解析+positions/{position}.json |

### 头部企业专家生成规则

**Phase 1 输入：**
```
用户岗位：B2B销售
行业：SaaS
意向公司：Salesforce
```

**Phase 2 圆桌讨论重点：**
- 该企业在同岗位的能力模型是什么？
- 该企业的工作方法论/思维框架有哪些可迁移？
- 用户与该企业同级别的人差距在哪？
- 该企业同岗位的日常工作节奏和思维方式

**Phase 3 Prompt 组合要点：**
- Layer 1 身份：{意向公司}的{岗位}高级专家
- Layer 2 背景故事：在{意向公司}工作N年，负责过核心业务
- Layer 4 知识：该企业的能力模型、晋升标准、面试要求
- Layer 5 约束：不说"建议去大厂"这种废话，给出具体差距和可迁移能力

**Phase 4 输出格式：**
```json
{
  "company": "Salesforce",
  "capability_model": ["能力1", "能力2"],
  "gap_analysis": {"gap": "...", "how_to_close": "..."},
  "transferable_methods": ["方法1", "方法2"],
  "daily_work_rhythm": "..."
}
```

### 岗位洞察生成规则

**Phase 1 输入：**
```
目标JD：[用户粘贴的JD文字]
用户画像：{portrait_json}
```

**Phase 2 圆桌讨论重点：**
- JD的真实要求是什么（不是字面意思）？
- 这个岗位在该公司/行业的定位？
- 用户与JD要求的匹配度？
- 用户的差距如何弥补？

**Phase 3 Prompt 组合要点：**
- Layer 1 身份：资深{行业}招聘专家+行业内部人
- Layer 4 知识：JD解析结果+行业岗位知识
- Layer 5 约束：不说"匹配度72分"这种伪精确数字，用定性评级

**Phase 4 输出格式：**
```json
{
  "jd解读": {"真实要求": "...", "隐含要求": "..."},
  "匹配评级": "强/中/弱",
  "维度评级": {"技能": "强/中/弱", "经验": "强/中/弱", "行业": "强/中/弱"},
  "差距清单": [{"gap": "...", "severity": "大/中/小", "how_to_close": "..."}],
  "优势清单": ["优势1", "优势2"]
}
```

### 与行业总监的区别

| 维度 | 行业总监 | 头部企业专家 | 岗位洞察 |
|------|---------|------------|---------|
| 知识注入 | positions/{position}.json | 公司公开信息+positions/{position}.json | JD解析+positions/{position}.json |
| 个性化程度 | 按岗位+行业 | 按岗位+行业+具体公司 | 按具体JD |
| 复用性 | 高（同岗位用户复用） | 低（一客一用） | 低（一客一用） |
| 缓存策略 | 可缓存（positions/） | 不缓存 | 不缓存 |
