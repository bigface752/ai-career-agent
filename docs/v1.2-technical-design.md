## 1. 问题陈述

### 1.1 现状

当前系统将「旅程」和「对话」拆成两个独立页面：

| 页面 | 职责 | 问题 |
|------|------|------|
| `/journey` | 展示 4 步进度条 + 开始/完成按钮 | 只是导航壳，不含实际内容 |
| `/dialogue` | 对话界面（仅 intake 步骤） | 脱离旅程上下文，用户不知道"我在哪" |
| `/portrait` | 画像查看（diagnosis 步骤） | 独立页面，需手动跳转 |
| `/evaluation` | 评估报告（coaching 步骤） | 独立页面，需手动跳转 |

**用户实际体验**：
```
/journey → 点"开始" → 跳 /dialogue → 聊完 → 回 /journey → 点"标记完成"
→ 点"下一步" → 跳 /portrait → 看完 → 回 /journey → 点"标记完成" → ...
```

每一步都是 **跳转 → 操作 → 返回 → 标记** 的循环。用户关心的是"我下一步该做什么"，不是"系统有几个模块"。

### 1.2 核心矛盾

旅程是线性流程，对话是交互载体——它们描述的是同一件事的两个维度。当前实现把维度当模块，导致：
- **认知断裂**：用户在 /dialogue 聊天时看不到旅程进度
- **操作冗余**：每步需要"开始 → 操作 → 返回 → 标记完成"四次点击
- **状态同步脆弱**：JourneySession 和 DialogueSession 独立维护，需要额外同步逻辑

## 2. 目标与非目标

### 2.1 目标

| # | 目标 | 可衡量标准 |
|---|------|-----------|
| G1 | 用户在一个页面完成整个旅程 | 从 intake 到 complete 不离开 /journey |
| G2 | 对话是旅程的交互方式 | 每个步骤的内容通过嵌入式对话完成 |
| G3 | 自动状态流转 | 对话满足条件后自动推进步骤，无需手动点击"完成" |
| G4 | 向后兼容 | 现有 /dialogue 页面保留，老用户已有的对话数据不丢失 |

### 2.2 非目标

| # | 非目标 | 原因 |
|---|--------|------|
| NG1 | 重写对话引擎 | 当前 slot-based 对话系统工作正常，只改嵌入方式 |
| NG2 | 合并数据库模型 | JourneySession + DialogueSession 各有职责，通过外键关联即可 |
| NG3 | 多模块对话切换 | V1.2 只处理 career 模块的旅程对话 |
| NG4 | 离线支持 | 网络依赖的 AI 对话无法离线 |

## 3. 架构设计

### 3.1 核心理念

**旅程即对话，步骤是对话的章节。**

```
┌─────────────────────────────────────────────┐
│  /journey (统一页面)                         │
│                                             │
│  ┌─ JourneyHeader ─────────────────────┐    │
│  │  步骤进度条 [1] [2] [3] [4]         │    │
│  └─────────────────────────────────────┘    │
│                                             │
│  ┌─ StepContent (动态渲染) ────────────┐    │
│  │                                     │    │
│  │  intake:    <DialoguePanel />       │    │
│  │  diagnosis: <PortraitPanel />       │    │
│  │  coaching:  <CoachingPanel />       │    │
│  │  complete:  <ReportPanel />         │    │
│  │                                     │    │
│  └─────────────────────────────────────┘    │
│                                             │
│  ┌─ ContextSidebar (可选) ─────────────┐    │
│  │  当前步骤说明 / 关键信息摘要        │    │
│  └─────────────────────────────────────┘    │
└─────────────────────────────────────────────┘
```

### 3.2 组件架构

```
src/
├── app/
│   └── journey/
│       └── page.tsx                  # 重写：统一容器
├── components/
│   └── journey/
│       ├── JourneyHeader.tsx         # 进度条（复用现有 JourneyProgress）
│       ├── StepContent.tsx           # 步骤内容路由（新增）
│       ├── DialoguePanel.tsx         # 嵌入式对话（从 /dialogue 提取）
│       ├── PortraitPanel.tsx         # 画像展示（从 /portrait 提取）
│       ├── CoachingPanel.tsx         # 行动建议（从 /evaluation 提取）
│       └── ReportPanel.tsx           # 最终报告（从 /dashboard 提取）
└── hooks/
    ├── useJourney.ts                 # 旅程状态管理（新增）
    └── useDialogue.ts                # 对话 hook（复用，微调）
```

### 3.3 状态流转

```
用户发送消息
    │
    ▼
DialoguePanel.sendMessage()
    │
    ▼
POST /api/dialogue/message  ← 现有 API，不改
    │
    ▼
响应包含 X-Dialogue-State header
    │
    ▼
DialoguePanel 检查 slot 填充率
    │
    ├── 未满 100% → 继续对话
    │
    └── 达到 100% → 触发自动推进
            │
            ▼
        useJourney.advanceStep()
            │
            ▼
        POST /api/journey/advance  ← 现有 API，增加 auto 参数
            │
            ▼
        StepContent 切换到下一个 Panel
```

### 3.4 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 对话组件复用 vs 重写 | 复用 useDialogue hook | 对话引擎工作正常，只改嵌入方式 |
| 自动推进 vs 手动确认 | 自动推进 + 3s 通知 | 减少操作步骤，通知让用户知道发生了什么 |
| 步骤内容内联 vs 路由 | 内联条件渲染 | 避免页面跳转，保持状态连续 |
| 旧 /dialogue 页面 | 保留但标记 deprecated | 向后兼容，避免破坏已有书签/链接 |

## 4. API 契约

### 4.1 新增 API

#### `POST /api/journey/advance` (修改)

现有 API 增加 `auto` 参数，支持自动推进场景。

**Request**:
```typescript
{
  version: number;       // 乐观锁版本号（必须）
  auto?: boolean;        // 是否自动推进（默认 false）
  reason?: string;       // 自动推进原因（日志用）
}
```

**Response (200)**:
```typescript
{
  session: {
    id: string;
    currentStep: "intake" | "diagnosis" | "coaching" | "complete";
    stepStatus: "pending" | "in_progress" | "completed";
    completedSteps: string[];
    version: number;
  };
  advanced: boolean;     // 是否实际推进了（auto 模式下可能因条件不满足而跳过）
}
```

**错误码**:
| Code | Status | 含义 |
|------|--------|------|
| `CONFLICT` | 409 | 版本号冲突，需刷新 |
| `CANNOT_ADVANCE` | 422 | 当前状态不允许推进 |
| `STEP_REQUIREMENTS_NOT_MET` | 422 | 步骤数据未满足推进条件 |

#### `GET /api/journey/status` (无变更)

现有 API 不变，返回完整的旅程状态 + 步骤进度。

#### `POST /api/dialogue/message` (无变更)

现有 API 不变。对话引擎通过 `X-Dialogue-State` header 返回状态更新。

### 4.2 复用 API

| API | 用途 | 变更 |
|-----|------|------|
| `POST /api/dialogue/start` | intake 步骤开始时自动调用 | 无 |
| `POST /api/dialogue/message` | 发送对话消息 | 无 |
| `GET /api/dialogue/status` | 恢复已有对话 | 无 |
| `POST /api/portrait/extract` | 画像提炼（对话触发） | 无 |
| `GET /api/evaluation/report` | 获取评估报告 | 无 |

### 4.3 数据流

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  前端         │     │  API 层       │     │  数据库       │
│              │     │              │     │              │
│ useJourney   │────▶│ /journey/*   │────▶│ JourneySession│
│ useDialogue  │────▶│ /dialogue/*  │────▶│DialogueSession│
│              │     │              │     │DialogueMessage│
│ StepContent  │◀────│ /journey/    │◀────│              │
│  .tsx        │     │  status      │     │              │
└──────────────┘     └──────────────┘     └──────────────┘
```

## 5. 数据模型

### 5.1 现有模型（不变）

```prisma
model JourneySession {
  id             String      @id @default(cuid())
  userId         String      @unique @map("user_id")
  currentStep    JourneyStep @map("current_step")
  stepStatus     StepStatus  @map("step_status")
  stepData       String?     @map("step_data")       // JSON
  completedSteps String?     @map("completed_steps") // JSON array
  version        Int         @default(0)
  createdAt      DateTime    @default(now()) @map("created_at")
  updatedAt      DateTime    @updatedAt @map("updated_at")
  expiresAt      DateTime?   @map("expires_at")
}

model DialogueSession {
  id              String    @id @default(cuid())
  userId          String    @map("user_id")
  module          String    // career / match / interview
  status          String    @default("active")
  slotState       String?   @map("slot_state")       // JSON
  recentWindow    String?   @map("recent_window")     // JSON
  initialFindings String?   @map("initial_findings")  // JSON
  roundNumber     Int       @default(0) @map("round_number")
  // ...
}
```

### 5.2 新增关联（可选增强）

在 `JourneySession` 中增加 `activeDialogueSessionId` 字段，建立与 `DialogueSession` 的显式关联：

```prisma
model JourneySession {
  // ... 现有字段 ...
  activeDialogueSessionId String? @map("active_dialogue_session_id")

  activeDialogueSession DialogueSession? @relation(fields: [activeDialogueSessionId], references: [id])
}
```

**收益**：
- intake 步骤可以快速找到关联的对话会话
- 避免通过 `userId + module` 查询的歧义（用户可能有多个历史会话）

**成本**：
- 一次 schema migration
- 需要更新 `createJourneySession` 和 `advanceJourney` 的逻辑

**决策**：V1.2 先不做 schema 变更，通过 `userId + module: "career" + status: "active"` 查询即可。如果后续需要多会话管理，再加字段。

### 5.3 状态数据结构

**JourneySession.stepData** (JSON):
```typescript
// intake 步骤
{
  dialogueSessionId: string;   // 关联的对话会话 ID
  slotCompletionRate: number;  // 0-1
  roundNumber: number;
}

// diagnosis 步骤
{
  portraitId: string;
  completionRate: number;
  keyFindings: string[];
}

// coaching 步骤
{
  evaluationId: string;
  actionItems: string[];
}
```

**JourneySession.completedSteps** (JSON):
```typescript
["intake", "diagnosis"]  // 已完成的步骤名数组
```

## 6. 迁移策略

### 6.1 渐进式迁移

不搞一次性重写，分三步走：

#### Phase 1: 嵌入对话组件（1-2 天）

1. 从 `/dialogue/page.tsx` 提取对话核心逻辑为 `DialoguePanel` 组件
2. 在 `/journey/page.tsx` 中根据 `currentStep` 条件渲染对应 Panel
3. 保留 `/dialogue` 页面，入口改为 `/journey` 的链接

**变更文件**：
| 文件 | 变更 |
|------|------|
| `src/components/journey/DialoguePanel.tsx` | 新增：从 dialogue/page.tsx 提取 |
| `src/components/journey/PortraitPanel.tsx` | 新增：从 portrait/page.tsx 提取 |
| `src/components/journey/CoachingPanel.tsx` | 新增：从 evaluation/page.tsx 提取 |
| `src/components/journey/ReportPanel.tsx` | 新增：从 dashboard/page.tsx 提取 |
| `src/components/journey/StepContent.tsx` | 新增：步骤内容路由 |
| `src/app/journey/page.tsx` | 重写：使用 StepContent 替代 JourneyStep |

#### Phase 2: 自动状态流转（1 天）

1. DialoguePanel 在 slot 填充率 100% 时自动调用 `advanceStep()`
2. 增加 3s toast 通知："信息采集完成，正在进入职业诊断..."
3. StepContent 切换动画（淡入淡出）

**变更文件**：
| 文件 | 变更 |
|------|------|
| `src/hooks/useJourney.ts` | 新增：旅程状态管理 hook |
| `src/components/journey/DialoguePanel.tsx` | 修改：增加自动推进逻辑 |
| `src/app/api/journey/advance/route.ts` | 修改：支持 auto 参数 |

#### Phase 3: 清理与优化（0.5 天）

1. `/dialogue` 页面添加 deprecation notice，引导到 `/journey`
2. 更新 Navbar 导航，统一入口为 `/journey`
3. 更新文档

### 6.2 数据兼容

| 场景 | 处理方式 |
|------|---------|
| 老用户有 DialogueSession 无 JourneySession | 自动创建 JourneySession，设为 intake + in_progress |
| 老用户有 JourneySession 在 diagnosis+ | 正常渲染对应 Panel |
| 老用户正在 /dialogue 页面 | 页面刷新后引导到 /journey |
| 新用户 | 直接走新流程 |

### 6.3 回滚方案

如果 V1.2 出现严重问题：
1. 前端回滚：`/journey/page.tsx` 恢复到旧版本（git revert）
2. 后端无需回滚：API 无破坏性变更
3. 数据无需迁移：未修改 schema

## 8. 风险与缓解

### 8.1 风险矩阵

| # | 风险 | 概率 | 影响 | 缓解措施 |
|---|------|------|------|---------|
| R1 | 对话组件从 /dialogue 提取后样式/行为不一致 | 中 | 中 | 提取时保持 CSS 类名一致，Diff 对比 |
| R2 | 自动推进时机不准（slot 100% 但用户还想补充） | 中 | 高 | 推进前显示 3s 通知 + "回到上一步"按钮 |
| R3 | JourneySession 和 DialogueSession 状态不同步 | 低 | 高 | 统一通过 /journey/status 查询，不缓存对话状态 |
| R4 | 老用户 /dialogue 页面书签失效 | 高 | 低 | 保留 /dialogue 页面，加 deprecation notice |
| R5 | 嵌入式对话在移动端体验差 | 中 | 中 | DialoguePanel 响应式设计，移动端全屏模式 |

### 8.2 性能风险

| 指标 | 当前值 | 目标值 | 监控方式 |
|------|--------|--------|---------|
| /journey 首屏加载 | ~200ms | <500ms | Sentry Performance |
| 对话消息响应 | <2s (streaming) | 不变 | X-Dialogue-State header |
| 自动推进延迟 | N/A | <3s | 前端日志 |

### 8.3 安全考量

| 项目 | 现状 | V1.2 变更 |
|------|------|----------|
| 认证 | JWT + cookie | 不变 |
| 速率限制 | 100次/分钟/IP | 不变 |
| 数据隔离 | userId 级别 | 不变 |
| XSS | React 默认转义 | 不变 |
