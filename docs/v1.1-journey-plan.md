# V1.1 用户旅程重构方案

## 一、问题诊断

当前V1的核心问题：**把"能力模块"当成了"用户步骤"**。

1. **首页是开发导航页**：直接暴露全部内部路由，没有产品介绍、用户旅程可视化、价值主张展示
2. **仪表盘完全未开发**：仅是占位组件，但PRD和user-flows都将其定义为登录后的默认入口
3. **模块割裂**：4个独立模块（模块零/一/二/三）导致用户在Landing Page就要做选择，但此时他们根本不知道自己需要什么
4. **模块零位置错误**：放在首页入口，但逻辑上应该在模块一判断为"不跳"后才出现

## 二、设计原则

### 第一性原理

- 用户不关心系统有几个模块，只关心"我下一步该做什么"
- 最短路径：建立认知 → 形成判断 → 行动建议
- 核心原则：用户不需要知道系统怎么工作，只需要跟着走

### 剃刀法则

- 能用最简方案解决的，不造复杂系统
- 砍掉所有不必要的元素
- 原方案：4个独立模块 × 多个入口 × 用户选择 = 复杂度高
- 新方案：3个顺序Step × 单一入口 × 系统引导 = 复杂度低

### 贝叶斯思维

- 先验：用户不知道自己需要什么（信息不足）
- 证据：用户上传简历、完成对话后，信息逐渐丰富
- 后验：基于足够信息，系统给出判断，用户知道下一步
- 新方案符合贝叶斯推理的信息积累过程：从不确定到确定，从模糊到清晰

## 三、技术方案结论

### 状态管理：自定义状态管理

**理由**：
- XState学习曲线高，对于当前场景过于复杂
- 我们的状态转换逻辑相对简单（3个Step，每个Step有明确的进入/退出条件）
- 符合剃刀法则：能用最简方案解决的，不造复杂系统

**实现**：
```typescript
// /src/lib/journey.ts
type Step = 'self-cognition' | 'judgment' | 'action'
type Status = 'pending' | 'in_progress' | 'completed'

interface JourneyState {
  currentStep: Step
  steps: Record<Step, Status>
  data: Record<string, any>
}
```

### 会话持久化：数据库（Turso）

**理由**：
- 已有Turso基础设施，不需要额外部署
- 用户换设备也能恢复会话
- 符合剃刀法则：复用现有基础设施

**实现**：
- 新增 `journey_sessions` 表
- 使用Prisma迁移

### Agent判断逻辑：规则引擎

**理由**：
- 判断条件明确（简历已解析、对话已进行、画像已建立等）
- 不需要AI的灵活性，规则引擎更可靠
- 符合剃刀法则：简单直接

**实现**：
```typescript
// /src/lib/journey-rules.ts
const rules = {
  'self-cognition': {
    canAdvance: (data) => data.resumeParsed && data.dialogueCompleted && data.portraitBuilt,
    nextStep: 'judgment'
  },
  'judgment': {
    canAdvance: (data) => data.roundtableCompleted && data.evaluationGenerated,
    nextStep: 'action'
  }
}
```

### 前端组件：最大化复用

**理由**：
- 已有Button、Card、ProgressBar、Input等基础组件
- 只需新建3个旅程专用组件
- 符合剃刀法则：复用现有组件

**实现**：
- 复用：`Button`、`Card`、`ProgressBar`、`Input`
- 新建：`JourneyProgress`、`JourneyStep`、`JourneyChat`

## 七、成功指标

### 核心指标

- 用户完成率：从首页到最终建议的完成比例
- 平均耗时：用户完成整个旅程的时间
- 回退率：用户回退到之前步骤的比例
- 放弃率：用户中途退出的比例

### 目标值

- 完成率：>70%（当前估计<30%，因为用户不知道该做什么）
- 平均耗时：<20分钟
- 回退率：<10%
- 放弃率：<20%
